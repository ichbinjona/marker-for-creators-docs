# Documentation Guidelines

These rules apply to all documentation pages for Marker for Creators.

## Tone & Mindset

- Write from the **user's perspective**. Focus on what they can do, not how we built it.
- Everything should feel **seamless and effortless**. The product "just works."
- Use emotional, benefit-driven intros for every feature section. Tell the user **why** this matters before explaining **how** it works.

## Never Expose Internal Architecture

- **Never mention** underlying technologies, libraries, or implementation details: no ffprobe, no Spotlight indexing, no Python bridge, no Lua scripts, no FCPXML, no XMEML, no OTIO format names, no EDL format names, no XML format names.
- **Exception:** Troubleshooting pages may reference internal systems (e.g., Spotlight indexing) **only** when the user encounters a specific error related to that system.
- Don't explain "how matching works" step-by-step. Say it works intelligently and automatically. Example: "The Companion learns your project structure over time and finds your footage faster with every session."
- Don't list supported video formats unless it's in a troubleshooting context.

## Formatting Rules

- **No em-dashes** (—) or en-dashes (–). Use regular hyphens (-) only.
- All content in **English**.
- No Free/Pro tier distinction in general docs. Don't mention subscription tiers unless directly relevant.
- **YouTube Mode** was renamed to **Review Mode**. Always use "Review Mode" in documentation.

## Feature Descriptions

- Don't over-explain. Keep it practical.
- Don't tell users what happens "behind the scenes" or "under the hood."
- For editing software integration: say "Press Play and your project opens in [Software]" - don't describe the file format or generation process.
- For DaVinci Resolve Free: the extra step is necessary but don't explain why (no "because the free version doesn't allow external scripts").

## Competitive Awareness

- Don't provide enough implementation detail for a competitor to replicate features.
- Automatic Footage Matching should be described abstractly: it works, it's smart, it learns. Not: "it checks these 5 directories in this order using these tools."
- Keep the "magic" feeling. Users should be impressed, not informed about the plumbing.

## Legacy Export

- Position Legacy Export as the **first-generation, manual workflow**. It's not "simpler" - it's more limited.
- The recommended workflow is always the Companion App with one-click import.
- Legacy Export exists for edge cases where the automatic workflow doesn't cover a specific use case.
